# SRS2 — Модуль DocuOrient: определение читаемой ориентации документа

## 0. Статус документа

| Поле | Значение |
|---|---|
| Модуль | `DocuOrient` |
| Версия спецификации | `0.1` |
| Статус | MVP-спецификация для реализации и приёмки |
| Источник | `SRS.md` в репозитории `AlexanderKuzikov/DocuOrient` |
| Автор оценки/версии | Hermes Agent |
| Дата | 2026-06-20 |

## 1. Назначение

`DocuOrient` — модуль для автоматического доворота уже выпрямленного grayscale-изображения юридического документа в читаемую ориентацию.

Модуль принимает на вход результат этапа `DocuDeskew` и определяет, требуется ли применить поворот на `0°`, `90°`, `180°` или `270°` по часовой стрелке, чтобы текст документа располагался сверху вниз и слева направо в нормальной читаемой ориентации.

Модуль не выполняет deskew, OCR, распознавание текста, классификацию документа или исправление качества скана.

## 2. Цели

### 2.1. Основные цели

1. Определить необходимый доворот документа без OCR и нейросетей.
2. Применить доворот к исходному изображению.
3. Вернуть итоговое изображение в формате PNG и метаданные решения.
4. Не выполнять доворот при низкой уверенности, чтобы не ухудшать уже корректный документ.

### 2.2. Нецели

Модуль не должен:

- выравнивать перекос страницы;
- распознавать текст;
- определять тип документа;
- восстанавливать качество скана;
- работать с цветными фотографиями как с основным сценарием;
- гарантировать корректность на произвольных изображениях без текста.

## 3. Область применения

### 3.1. Поддерживаемый сценарий

Поддерживается конвейер:

```text
DocuDeskew → DocuOrient → OCR / дальнейшая обработка
```

На вход поступает изображение, у которого уже исправлен геометрический перекос, но неизвестна читаемая ориентация.

### 3.2. Типовые документы

Модуль ориентирован на:

- сканы юридических документов;
- документы с печатным русским текстом;
- документы с полями, шапками, таблицами, реквизитами;
- grayscale PNG-изображения.

### 3.3. Ограничения MVP

Для MVP модуль не обязан гарантировать корректную ориентацию:

- полностью пустых страниц;
- страниц с минимальным количеством текста;
- рукописных документов без печатного текста;
- фотографий документов с сильным шумом, бликами, тенью;
- документов с нестандартной вертикальной вёрсткой;
- смешанных ориентаций внутри одного изображения;
- изображений с сильным кадрированием полей.

## 4. Термины

| Термин | Значение |
|---|---|
| Deskew | Исправление геометрического перекоса страницы |
| Orientation | Угловая ориентация документа: `0°`, `90°`, `180°`, `270°` |
| Rotation step | Угол доворота, который нужно применить к входному изображению |
| Projection | Сумма тёмных пикселей по строкам изображения после бинаризации |
| Texture score | Мера неоднородности проекции, показывающая наличие строк текста |
| Brightness asymmetry | Разница средней яркости верхней/нижней или левой/правой области |
| Low confidence | Состояние, при котором алгоритм не уверен в довороте и оставляет изображение без изменений |

## 5. Внешний интерфейс

### 5.1. Основная функция

```typescript
export async function orient(
  input: Buffer | Uint8Array,
  options?: OrientationOptions
): Promise<OrientationResult>;
```

### 5.2. Входные данные

| Параметр | Тип | Обязательный | Описание |
|---|---|---:|---|
| `input` | `Buffer \| Uint8Array` | Да | PNG-изображение документа |
| `options` | `OrientationOptions` | Нет | Настройки алгоритма |

### 5.3. Типы

```typescript
type RotationStep = 0 | 90 | 180 | 270;

interface OrientationOptions {
  /**
   * Использовать Otsu-порог.
   * Если false, используется fixedThreshold.
   */
  otsuThreshold?: boolean;

  /**
   * Фиксированный порог бинаризации при otsuThreshold = false.
   * Диапазон: 0–255.
   */
  fixedThreshold?: number;

  /**
   * Минимальный нормализованный texture score.
   * Ниже этого значения документ считается нетекстовым или недостаточно информативным.
   */
  minTextureScore?: number;

  /**
   * Минимальная уверенность для выполнения доворота.
   * Если confidence < minConfidence, модуль возвращает исходное изображение и rotationStep = 0.
   */
  minConfidence?: number;

  /**
   * Коэффициент преимущества лучшей оси строк.
   * Используется для выбора между горизонтальной и вертикальной ориентацией.
   */
  axisScoreRatio?: number;

  /**
   * Минимальная разница яркости между верхней/нижней или левой/правой областью.
   */
  brightnessMargin?: number;

  /**
   * Доля изображения, исключаемая из расчётов по внешнему краю.
   * Рекомендуется 0.05–0.10 для уменьшения влияния рамок, шума и артефактов скана.
   */
  ignoreBorderRatio?: number;

  /**
   * Вернуть служебные данные для отладки.
   */
  returnDebug?: boolean;
}

interface OrientationResult {
  /**
   * Применённый доворот в градусах по часовой стрелке.
   */
  rotationStep: RotationStep;

  /**
   * Финальное grayscale PNG-изображение.
   * При низком confidence совпадает со входным изображением.
   */
  orientedImage: Buffer;

  /**
   * Уверенность алгоритма в диапазоне 0–1.
   */
  confidence: number;

  /**
   * Тип решения.
   */
  decision: 'rotated' | 'unchanged' | 'low_confidence';

  /**
   * Служебные данные, если returnDebug = true.
   */
  debug?: OrientationDebug;
}

interface OrientationDebug {
  scores: {
    rotation0: number;
    rotation90: number;
    rotation180: number;
    rotation270: number;
  };

  axis: 'horizontal' | 'vertical' | 'unknown';

  brightness: {
    top?: number;
    bottom?: number;
    left?: number;
    right?: number;
  };

  thresholds: {
    minTextureScore: number;
    minConfidence: number;
    brightnessMargin: number;
  };
}
```

### 5.4. Выходные данные

Модуль всегда возвращает `Promise<OrientationResult>`.

Если вход корректен, функция не должна выбрасывать ошибку из-за отсутствия уверенности. В таком случае возвращается:

```typescript
{
  rotationStep: 0,
  orientedImage: input,
  confidence: number,
  decision: 'low_confidence'
}
```

### 5.5. Ошибки

| Ошибка | Когда возникает |
|---|---|
| `InvalidInputError` | `input` отсутствует, не является `Buffer` или `Uint8Array` |
| `UnsupportedFormatError` | вход не является PNG |
| `UnsupportedImageError` | изображение не 8-bit grayscale или не может быть приведено к нему |
| `EmptyImageError` | изображение пустое, имеет нулевую ширину/высоту или не содержит полезных пикселей |
| `ProcessingError` | внутренняя ошибка обработки изображения |

## 6. Алгоритм обработки

### 6.1. Общая схема

```text
1. Валидация входного PNG
2. Приведение к 8-bit grayscale
3. Бинаризация
4. Расчёт проекций для 0°, 90°, 180°, 270°
5. Расчёт texture score для каждой ориентации
6. Выбор оси строк: горизонтальная или вертикальная
7. Выбор направления внутри пары: 0/180 или 90/270
8. Расчёт confidence
9. Применение доворота или возврат исходного изображения
10. Возврат результата
```

### 6.2. Валидация

Перед обработкой необходимо проверить:

- вход является `Buffer` или `Uint8Array`;
- данные читаются как PNG;
- изображение имеет ненулевую ширину и высоту;
- изображение является 8-bit grayscale или может быть безопасно приведено к 8-bit grayscale;
- изображение не является полностью пустым.

Если проверка не пройдена, функция возвращает соответствующую ошибку.

### 6.3. Бинаризация

Если `otsuThreshold !== false`:

1. Вычислить Otsu-порог для grayscale-изображения.
2. Пиксели темнее порога считать текстом.
3. Пиксели светлее порога считать фоном.

Если `otsuThreshold = false`:

1. Использовать `fixedThreshold`.
2. Значение по умолчанию: `128`.
3. Диапазон: `0–255`.

После бинаризации текст должен быть представлен как `1`, фон как `0`.

### 6.4. Расчёт проекций

Для каждой кандидатной ориентации `R ∈ {0, 90, 180, 270}`:

1. Повернуть бинарное изображение на `R` градусов по часовой стрелке.
2. Для каждой строки `y` посчитать сумму единиц:
   ```text
   projection[y] = sum(binaryRow[y])
   ```
3. Нормализовать проекцию по ширине изображения:
   ```text
   density[y] = projection[y] / width
   ```
4. Вычислить texture score:
   ```text
   score[R] = stdDev(density)
   ```

Нормализация по ширине обязательна, чтобы score был сопоставим между ориентациями и размерами изображения.

### 6.5. Выбор оси строк

1. Сравнить лучшие score для горизонтальной пары и вертикальной пары:
   ```text
   horizontalScore = max(score[0], score[180])
   verticalScore   = max(score[90], score[270])
   ```

2. Если оба значения ниже `minTextureScore`:
   ```text
   axis = 'unknown'
   decision = 'low_confidence'
   rotationStep = 0
   ```

3. Если:
   ```text
   horizontalScore >= verticalScore * axisScoreRatio
   ```
   то:
   ```text
   axis = 'horizontal'
   candidatePair = [0, 180]
   ```

4. Иначе:
   ```text
   axis = 'vertical'
   candidatePair = [90, 270]
   ```

Значение `axisScoreRatio` по умолчанию: `1.05`.

### 6.6. Выбор направления для горизонтальной пары

Для пары `0°/180°` анализируется кандидатная ориентация, соответствующая текущему положению страницы.

1. Разделить изображение на верхнюю, среднюю и нижнюю трети.
2. Исключить внешнюю границу согласно `ignoreBorderRatio`.
3. Посчитать среднюю яркость в исходной grayscale-шкале:
   - `topBrightness`;
   - `bottomBrightness`.
4. Если:
   ```text
   topBrightness >= bottomBrightness + brightnessMargin
   ```
   то текст, вероятно, уже расположен нормально:
   ```text
   rotationStep = 0
   ```
5. Иначе:
   ```text
   rotationStep = 180
   ```

Для MVP верхняя область считается более светлой из-за полей, шапки или меньшего количества текста.

### 6.7. Выбор направления для вертикальной пары

Для пары `90°/270°`:

1. Рассмотреть два кандидатных положения: `90°` и `270°`.
2. Для каждого кандидата повернуть бинарное изображение на соответствующий угол.
3. Разделить кандидатное изображение на левую, среднюю и правую трети.
4. Исключить внешнюю границу согласно `ignoreBorderRatio`.
5. Посчитать среднюю яркость:
   - `leftBrightness`;
   - `rightBrightness`.
6. Выбрать кандидат, у которого левая область светлее правой:
   ```text
   leftBrightness >= rightBrightness + brightnessMargin
   ```
7. Если ни один кандидат не даёт уверенного преимущества, вернуть `low_confidence`.

Примечание: для юридических русскоязычных документов вертикальная ориентация текста считается редким edge case. Если статистика показывает низкую точность, вертикальный режим можно отключить через опцию или пометить как экспериментальный.

### 6.8. Расчёт confidence

Минимальная модель уверенности:

```text
textureConfidence =
  max(horizontalScore, verticalScore) / max(minTextureScore, epsilon)

axisConfidence =
  max(horizontalScore, verticalScore) /
  max(min(horizontalScore, verticalScore) + epsilon, epsilon)

brightnessConfidence =
  abs(topBrightness - bottomBrightness) / 255
  или
  abs(leftBrightness - rightBrightness) / 255

confidence = clamp(
  0.45 * textureConfidence +
  0.35 * axisConfidence +
  0.20 * brightnessConfidence,
  0,
  1
)
```

Где `epsilon = 0.0001`.

Если `confidence < minConfidence`, модуль не выполняет доворот.

Значение `minConfidence` по умолчанию: `0.55`.

### 6.9. Выполнение доворота

Если `decision !== 'low_confidence'`:

1. Применить к исходному изображению:
   ```text
   sharp(input).rotate(rotationStep, { background: 255 }).png()
   ```
2. Вернуть итоговое PNG.

Если `decision === 'low_confidence'`:

1. Не выполнять доворот.
2. Вернуть входное изображение.
3. Установить:
   ```text
   rotationStep = 0
   ```

## 7. Настройки по умолчанию

```typescript
const DEFAULT_OPTIONS: Required<Pick<
  OrientationOptions,
  | 'otsuThreshold'
  | 'fixedThreshold'
  | 'minTextureScore'
  | 'minConfidence'
  | 'axisScoreRatio'
  | 'brightnessMargin'
  | 'ignoreBorderRatio'
  | 'returnDebug'
>> = {
  otsuThreshold: true,
  fixedThreshold: 128,
  minTextureScore: 0.015,
  minConfidence: 0.55,
  axisScoreRatio: 1.05,
  brightnessMargin: 3,
  ignoreBorderRatio: 0.05,
  returnDebug: false
};
```

Примечание: `minTextureScore` заменён с абсолютного значения `5.0` на нормализованный score. Это делает порог менее зависимым от размера изображения.

## 8. Нефункциональные требования

| Требование | Значение |
|---|---|
| Язык | TypeScript / Node.js |
| Runtime | Node.js 20+ |
| Основная зависимость | `sharp` |
| OCR | Не используется |
| Нейросети | Не используются |
| Сетевые вызовы | Запрещены |
| Детерминированность | Один и тот же input должен давать один и тот же output |
| Формат выхода | PNG, 8-bit grayscale |

| Логирование | Без логирования содержимого изображения и пользовательских данных |

## 9. Критерии приёмки

### 9.1. Функциональная приёмка

Модуль считается прошедшим функциональную приёмку, если:

| № | Критерий | Требование |
|---:|---|---|
| 1 | Корректный ввод | Для корректного PNG возвращается `OrientationResult` |
| 2 | Поворот 0° | Для уже читаемого документа возвращается `rotationStep = 0` |
| 3 | Поворот 90° | Для документа, повернутого на 90°, возвращается применимый доворот |
| 4 | Поворот 180° | Для перевёрнутого документа возвращается `rotationStep = 180` |
| 5 | Поворот 270° | Для документа, повернутого на 270°, возвращается применимый доворот |
| 6 | Низкая уверенность | При недостатке текста модуль не выполняет рискованный доворот |
| 7 | Ошибки | Некорректный ввод возвращает понятную ошибку |
| 8 | Формат выхода | `orientedImage` является PNG-буфером |
| 9 | Детерминированность | Повторный вызов на том же input даёт тот же результат |

### 9.2. Метрики качества

Минимальный MVP-датасет:

| Параметр | Требование |
|---|---|
| Количество изображений | ≥ 200 |
| Классы ориентации | 0°, 90°, 180°, 270° |
| Баланс классов | Не хуже 70/30 между самым частым и самым редким классом |
| Типы документов | Не менее 3 типов юридических документов |
| Разрешения | Не менее 2 разрешений |
| Edge cases | Пустые страницы, мало текста, таблицы, штампы, подписи |

Целевые метрики:

| Метрика | MVP-требование |
|---|---:|
| Accuracy по всему датасету | ≥ 95% |
| Accuracy по 0°/180° | ≥ 96% |
| Accuracy по 90°/270° | ≥ 90% или режим помечается как experimental |
| False rotate на пустых/малотекстовых страницах | ≤ 2% |
| p95 времени обработки | ≤ 100 мс |
| Доля low-confidence на валидных документах | ≤ 10% |

Если целевые метрики не достигнуты, модуль не считается готовым к production.

## 10. Тест-план

### 10.1. Unit-тесты

| Тест | Проверяет |
|---|---|
| Otsu threshold | Порог вычисляется корректно |
| Fixed threshold | Порог `128` используется при отключённом Otsu |
| Projection | Суммы по строкам считаются корректно |
| Texture score | Score растёт при наличии строк текста |
| Rotation mapping | 0/90/180/270 соответствуют повороту по часовой стрелке |
| Confidence | Confidence находится в диапазоне 0–1 |
| Low confidence | При слабом сигнале возвращается `decision = 'low_confidence'` |

### 10.2. Integration-тесты

| Тест | Проверяет |
|---|---|
| PNG grayscale | Корректная обработка базового входа |
| PNG RGB | Корректное приведение или понятная ошибка |
| Invalid PNG | Возвращается `UnsupportedFormatError` |
| Empty image | Возвращается `EmptyImageError` |
| 180° document | Применяется `rotationStep = 180` |
| 90° document | Применяется корректный доворот |
| 270° document | Применяется корректный доворот |
| Blank page | Не выполняется ложный доворот |
| Debug mode | Возвращаются score, axis и brightness |

### 10.3. Golden fixtures

Для каждого класса ориентации должны быть фиксированные изображения:

```text
fixtures/
  orientation-000.png
  orientation-090.png
  orientation-180.png
  orientation-270.png
  blank.png
  low-text.png
  table.png
  stamp.png
```

Для каждого fixture фиксируется ожидаемый результат:

```json
{
  "rotationStep": 180,
  "decision": "rotated",
  "minConfidence": 0.55
}
```

Benchmark должен измерять:

- время обработки одного изображения;

## 11. Риски и ограничения

| Риск | Вероятность | Влияние | Митигация |
|---|---:|---:|---|
| Яркость верха/низа не отражает читаемость | Средняя | Высокое | confidence, no-rotate при низкой уверенности |
| Штампы/подписи искажают асимметрию | Средняя | Высокое | exclude border, calibration на датасете |
| Таблицы дают ложные строки | Средняя | Среднее | texture score по нескольким ориентациям |
| 90°/270° менее надёжны | Средняя | Среднее | отдельная метрика и experimental-статус |
| Пустые страницы могут быть повернуты | Низкая | Высокое | low-confidence no-op |
| Производительность зависит от размера | Средняя | Среднее | ограничение размера и benchmark |

| Нужна ли поддержка вертикального текста в MVP? | Усложняет алгоритм и снижает точность | Для юридических документов — experimental |
| Какой reference CPU использовать для 100 мс? | Без hardware требование не проверяемо | Зафиксировать в CI/benchmark |
| Кто предоставляет acceptance dataset? | Без датасета нельзя подтвердить 95–98% | Создать минимальный набор fixtures и production-like dataset |
| Допустим ли RGB input? | Пользователи могут подавать не только grayscale | Для MVP: приводить к grayscale, но документировать |
| Нужен ли fallback на OCR? | Может повысить точность | Не включать в MVP, оставить как отдельный модуль |

## 13. Definition of Done

Модуль считается готовым к MVP, если:

- [ ] реализована функция `orient(input, options)`;
- [ ] реализованы все типы из раздела 5;
- [ ] реализована валидация входа;
- [ ] реализован алгоритм с confidence;
- [ ] при низком confidence доворот не выполняется;
- [ ] добавлены unit-тесты;
- [ ] добавлены integration-тесты;
- [ ] добавлены golden fixtures;
- [ ] добавлен benchmark;
- [ ] README обновлён под фактический API;
- [ ] acceptance metrics проверены на датасете;
- [ ] ограничения модуля явно описаны в документации.
