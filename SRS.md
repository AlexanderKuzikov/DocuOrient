# Модуль «Orientation» — доворот до читаемой ориентации

### 1. Назначение
Принять на вход уже выровненное по краям grayscale-изображение документа (результат Deskew) и определить, требуется ли доворот на 0°, 90°, 180° или 270° для приведения текста в читаемую ориентацию. Выполнить соответствующий поворот.

### 2. Входные данные
- `deskewedImage` — буфер выровненного изображения в формате PNG, **8-бит grayscale**.
- `options` (необязательный):
  - `otsuThreshold` — использовать ли Otsu для бинаризации внутри (по умолчанию `true`).
  - `minTextureScore` — минимальное значение СКО проекции, ниже которого документ считается нетекстовым и не поворачивается (по умолчанию 5.0 при высоте ~2000 px, должно калиброваться).

### 3. Выходные данные
Promise, разрешающийся объектом:
```typescript
{
  rotationStep: 0 | 90 | 180 | 270;  // применённый доворот
  orientedImage: Buffer;              // финальное grayscale PNG-изображение
}
```

### 4. Алгоритм обработки

#### 4.1. Бинаризация
- Применить Otsu-порог для получения бинарного изображения (текст — чёрный `0`, фон — белый `255`). Если `otsuThreshold = false`, использовать фиксированный порог `128`.
- Инвертировать, чтобы текст стал единицами (для удобства проекции).

#### 4.2. Оценка текстурности по горизонтальной проекции
- Для исходного изображения и для повёрнутого на 90° (по часовой) вычислить горизонтальную проекцию: для каждой строки `y` посчитать сумму инвертированных пикселей (т.е. вес текста).
- Для каждого варианта вычислить СКО (среднеквадратичное отклонение) проекции. Обозначим `score0` для 0°, `score90` для 90°.
- Если оба значения меньше `minTextureScore`, считать, что текст отсутствует, и установить `rotationStep = 0`.

#### 4.3. Определение оси строк
- Если `score0 > score90`, значит строки текста идут горизонтально (документ в паре 0°/180°).
- Иначе — вертикально (пара 90°/270°).

#### 4.4. Уточнение внутри пары (верх/низ или право/лево)
**Для пары 0°/180°:**
- Разделить изображение на три горизонтальные полосы (верхняя треть, средняя, нижняя).
- Посчитать среднюю яркость (в исходной шкале 0–255, до инверсии) в верхней и нижней трети. В читаемом положении верх обычно светлее (поля, шапка), низ темнее (плотный текст, реквизиты).
- Если `topBrightness > bottomBrightness` — оставляем 0°, иначе `rotationStep = 180`.

**Для пары 90°/270°:**
- Разделить изображение на три вертикальные полосы.
- Сравнить среднюю яркость левой и правой трети. Для документов с левым полем обычно левая треть светлее. Если `leftBrightness > rightBrightness` — это положение, близкое к 90° (или 270°, зависит от того, какой поворот дал вертикальные строки). Точный угол подбирается логически: вычислить `rotationStep` так, чтобы после доворота текст встал правильно.

#### 4.5. Выполнение поворота
- Используя `sharp.rotate(rotationStep, { background: 255 })`, довернуть исходное `deskewedImage`.
- Результат сохранить как PNG-буфер.

### 5. Требования к реализации
- **Зависимости:** `sharp`.
- **Производительность:** ≤ 100 мс на документ.
- **Устойчивость:** при низком текстурном скоре не выполнять доворот, возвращать исходное изображение и `rotationStep = 0`.

### 6. Критерии приёмки
- На наборе выровненных документов с текстом на русском языке, повёрнутых случайно на 0, 90, 180, 270 градусов:
  - Правильный `rotationStep` определяется в ≥ 98% случаев.
  - Итоговый текст читается в нормальной ориентации (визуальная проверка).
  - Время работы модуля ≤ 100 мс.
